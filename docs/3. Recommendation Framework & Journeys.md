# Recommendation Framework & Journeys
## Defining What a Recommendation Is and How Users Experience It

**Version:** 1.0

**Status:** Draft

**Priority:** Critical

**Consolidates:** Chapters 6 & 8

---

# 1. What Is a Recommendation?

> A context-aware proposal for one or more meals that maximizes expected household satisfaction while respecting all known constraints.

Key words:

- **Context-aware** — depends on today's context, not historical popularity.
- **Proposal** — the AI suggests; the household decides. Never authoritative.
- **Household satisfaction** — not calories, not recipe ratings, not internet popularity.
- **Constraints** — inventory, restrictions, budget, weather, effort, history, upcoming meals, seasonality, time.

---

# 2. The Recommendation Object

Every recommendation can be thought of as a structured object:

```
Recommendation
  Primary Meal
  Side Dish
  Bread/Rice
  Accompaniment
  Estimated Preparation Time
  Confidence Score
  Reasons (explanation)
  Rejected Alternatives
  Future Meal Impact
```

Notice: recipe instructions are missing. Recipes are secondary. The recommendation is primarily a **decision object**.

# 4. Inputs

The recommendation engine consumes signals from multiple domains:

**Household:** Members, preferences, restrictions, allergies, acceptance history.

**Inventory:** Vegetables, grains, spices, frozen food, prepared ingredients, leftovers.

**Context:** Time, weather, season, festival, day of week, upcoming travel, guests.

**Behaviour:** Recent meals, ignored/accepted recommendations, voice conversations, thumbs up/down, meal photographs.

**AI-Derived:** Household personality, household rhythm, ingredient repetition, cooking effort preference, confidence.

---

# 5. Outputs

Every recommendation should produce more than a dish name:

- Primary recommendation
- Why (explanation)
- Alternatives
- Preparation estimate
- Ingredient gaps
- Future planning impact

Example:

```
Recommended: Rajma Chawal

Why?
  Rajma has not been cooked in 18 days.
  You soaked Rajma yesterday.
  Weather is rainy.
  Children usually enjoy Rajma.

Alternatives: Vegetable Pulao, Chole

Rejected: Paneer — cooked two days ago.
```

---

# 6. Recommendation Confidence

Our AI should know when it is uncertain.

Confidence depends on: household understanding, inventory quality, preference quality, behaviour history, context quality.

**High confidence (92%):** one strong recommendation, minimal explanation.

**Low confidence (38%):** more alternatives, lightweight diagnostic question.

See *AI, Learning & Onboarding* §6 for the full AI confidence vs. user confidence distinction.

# 7. Recommendation Categories

A recommendation may belong to multiple categories:

Quick Meal · Comfort Food · Weekend Special · Festival Meal · Healthy Meal · Budget Meal · Leftover Meal · Guest Meal · Lunchbox Friendly · Low Effort · High Protein · Seasonal

# 8. Recommendation Horizon

Every recommendation influences future recommendations:

```
Today: Cook extra rice
  → Tomorrow: Fried Rice
    → Day After: Vegetable Soup
```

The recommendation engine operates on a **planning horizon** rather than isolated meals.

---

# 9. Explainability

Every recommendation must answer: Why this? Why today? Why not something else?

**Progressive Explanation Model** (Blueprint Principle 9):

- Default: one-line rationale. *"Rajma — rainy weather, soaked yesterday."*
- On demand: full reasoning with rejected alternatives.
- New users: more explanation to build trust.
- Experienced users: less explanation to reduce noise.
- Rejected recommendations: always show full reasoning.

---

# 10. Recommendation Diversity

Without diversity, users experience monotony. The engine should deliberately balance:

Ingredient diversity · Cuisine diversity · Cooking method diversity · Colour diversity · Texture diversity · Protein diversity · Vegetable diversity · Seasonal diversity

The goal is **balanced variety**, not random variety.

---

# 11. Ranking Strategy

Recommendations combine multiple weighted scores: Availability, Preference, Variety, Effort, Nutrition, Budget, Waste reduction, Seasonality, Household rhythm, Future planning.

The precise mathematical model is defined in *AI, Learning & Onboarding* §5.

---

# 12. Recommendation Lifecycle

```
Candidate Generation → Constraint Filtering → Ranking → Explanation Generation
    → Presentation → Feedback Collection → Learning → Future Recommendations
```

Recommendation is a continuous learning loop.

---

# 13. The Core User Journey

Everything in the application supports one fundamental journey:

```
Uncertainty → Need a meal → Receive recommendation → Understand reasoning
  → Choose confidently → Cook → Lightweight feedback → AI learns
    → Better recommendation tomorrow
```

---

# 14. Key Journeys

### A. First-Time User
Goal: "I just want help deciding dinner." Not: "I want to spend twenty minutes configuring preferences." The user should receive a plausible recommendation within two minutes of installing.

### B. Returning User
Goal: "Just tell me what I should cook." The recommendation should feel almost obvious. Users should increasingly feel: "That's exactly what I was going to cook."

### C. Rejected Recommendation
The app should understand *why*. "Too much effort?" / "Missing ingredients?" / "Not in the mood?" / "Already cooked recently?" Every rejection is valuable training data.

### D. Decision Paralysis
The highest-value moment. Be decisive. Fewer options, not more. Example:

```
Tonight I'd recommend: Rajma Chawal
Why? • Rainy weather • Rajma already soaked • Last cooked 19 days ago • Kids rate it highly
Two alternatives if you're not in the mood.
```

### E. Voice-First Interaction
Cook's hands are occupied. "What should I make with potatoes and spinach?" AI: "Aloo Palak. Hasn't been cooked in two weeks, pairs well with yesterday's leftover dal."

### F. Leftover Discovery
Photo upload, voice, or inventory → AI converts leftovers into opportunity. Yesterday's Dal → Dal Paratha → Lunch solved.

### G. Grocery Planning
Instead of a generic list: "If you buy these five items, your meal options for the next four days increase significantly." The shopping list becomes recommendation-aware.

### H. Festival / Special Occasion
Recognize the event. Temporarily adjust recommendations. Learn recurring traditions rather than treating them as anomalies.

### I. Busy Week
Detect late cooking, repeated quick meals, reduced interaction. Automatically switch to Quick Mode — lower effort, more leftovers, fewer experiments. No configuration required.

### J. Holiday / Vacation
Recognize travel, restaurant-heavy periods. Avoid learning incorrect long-term behaviour from temporary routines.

---

# 15. Decision States

Every journey begins with a mental state:

"I have no idea." / "I know the vegetable but not the dish." / "I know the dish but not the sides." / "I want something quick." / "I want to use leftovers." / "I want to impress guests."

Recognizing these states is often more important than recognizing ingredients.

---

# 16. The Ideal Daily Experience

**5:45 PM** — Notification: "Based on today's weather, your inventory and what you've cooked recently, I recommend Palak Paneer with Jeera Rice. ~35 minutes. Shall I show alternatives?"

The user spends less than thirty seconds making the decision. Mission accomplished.

# 17. Measuring Quality

| Metric | What It Measures |
|--------|-----------------|
| Recommendation Acceptance Rate | Are recommendations right? |
| Decision Duration | Are we reducing decision time? |
| Recommendation Override Rate | How often do users change our suggestion? |
| Recommendation Confidence | How certain is the AI? |
| Household Satisfaction | Are families happy with what they cooked? |
| Meal Diversity | Are we preventing monotony? |
| Leftover Utilization | Are we reducing waste? |
| Daily Return Rate | Is the product becoming a habit? |
| Voice Interaction Success | Is voice working? |
| Explanation Viewed Rate | Are users engaging with reasoning? |

---

# 18. Feedback

Every recommendation produces valuable learning:

**Positive:** Cooked, Saved, Liked, Shared.

**Negative:** Ignored, Rejected, Modified, Skipped.

Every outcome improves the Household Twin.

---

# 19. Common Failure Modes

The app should recognize when a journey has broken down: user keeps scrolling, repeated rejections, no recommendation selected, immediate app exit, repeated manual searches. These indicate insufficient confidence. The product should adapt.

# Cross-Journey Principles

1. Reduce thinking before reducing clicks.
2. Explain recommendations.
3. Learn continuously.
4. Respect household traditions.
5. Never overwhelm users with options.
6. Every interaction should improve tomorrow's recommendation.
