# Design Revision: Meal Assistant v2

**Date:** 2025-07-04
**Status:** Active — supersedes MVP design where contradicted

---

## Philosophy Revision

### ~~D-039: "Onboarding captures no more than three required data points"~~

**Revised:** Onboarding captures enough to make Day 1 feel personal: household profile, key likes/dislikes, and a snapshot of what's available. This is 5 short steps (~2 minutes), not 3 fields. The trade-off is 60 extra seconds of taps for recommendations that feel relevant from the first suggestion.

**Rationale:** The cold start is a trust problem (doc 5 §1). Generic recommendations from "population priors" don't build trust — they feel like a random recipe site. A few tap-based preference captures during onboarding produce a Day 1 recommendation that feels like it came from someone who knows your family. That's what builds trust.

### ~~D-042: "Progressive profiling through natural interactions, not forms"~~

**Revised:** Progressive profiling is the steady-state strategy. Onboarding is NOT progressive profiling — it's the initial calibration. Conflation of the two led to an anemic onboarding that produces poor Day 1 experiences.

**The distinction:**
- **Onboarding** = one-time calibration (household, preferences, baseline inventory, recent meals)
- **Progressive profiling** = ongoing learning from behavior (accept/reject patterns, ingredient frequency, rhythm detection)

Both are needed. They are not the same thing.

### ~~D-029 (implied): "The seed DB is the universe of recommendable dishes"~~

**Revised:** The seed DB is a starting corpus, not a closed universe. Users can log any dish via free text or photo. Custom dishes are created on-the-fly with AI-inferred metadata (ingredients, cuisine, category). The recommendation engine includes custom dishes in its candidate pool.

### ~~Photo recognition is Phase 4 (doc 0 §18)~~

**Revised:** Photo recognition is Phase 1. It is the lowest-friction meal logging method and the foundation of the learning loop. Without easy logging, the learning loop doesn't have data to learn from.

## Revised Onboarding Flow (5 Steps)

### Step 0: Welcome (unchanged)
- "Meal Assistant — What should we cook tonight?"
- "Takes about 2 minutes"

### Step 1: Household Basics (unchanged)
- Household name, size, cuisine region, diet type
- All tap-based, no typing beyond the name

### Step 2: Family + Preferences (EXPANDED)
- Add members with names (existing)
- For each member: quick toggle chips for common likes and dislikes
  - "Loves" chips: Paneer, Dal, Bhindi, Rajma, Chicken, Fish, Rice, Roti, etc.
  - "Won't eat" chips: same list, different selection
  - These are suggestions, not exhaustive — user taps what applies
- Common allergens as toggle chips (not free text): Nuts, Dairy, Shellfish, Gluten, Soy, Eggs
- This seeds the preference model and restriction engine immediately

### Step 3: What's in your kitchen? (NEW — moved from post-onboarding)
- The existing ingredient checklist (16 items, 4 categories)
- Framed as "What do you usually have at home?" (baseline pantry)
- Quick toggle, 15 seconds
- This prevents Day 1 recommendations that require ingredients the household doesn't have

### Step 4: What did you cook recently? (NEW)
- "To get started, what did you cook in the last few days?"
- **Three input modes:**
  1. **Free text**: "Dal chawal, paneer butter masala, bhindi" (comma or newline separated)
  2. **Photo**: "Snap a photo of what you made recently" → AI recognizes dish(es)
  3. **Skip**: "I'll log meals as I go"
- This seeds the meal history so:
  - The repetition check (D-009) works from Day 1
  - The recommendation engine has context from the first recommendation
  - The user sees immediate value (history is populated, not empty)

## Revised Home Screen

### Primary View (unchanged core, enhanced)
- Recommendation card with explanation + confidence (existing)
- Accept / Not today buttons (existing)
- **NEW**: Prominent "Log a meal" button (not hidden in a menu)
  - This is a first-class action, equal in weight to accepting/rejecting
  - Because logging what you actually cooked is more important than accepting a suggestion

### After Accepting
- **NEW**: Optional photo step — "Snap a photo of what you cooked?" (can skip)
  - This feeds the learning loop and the photo recognition training data
  - Photo is processed (dish recognition confirmation) then deleted per D-047
- Meal is logged immediately on accept; photo is a bonus, not a gate

### After Rejecting
- **NEW**: Instead of just showing next alternative, offer:
  - "Cooked something else?" → free text / photo input
  - "Show me alternatives" → existing alternative flow
  - This captures what they actually cooked — critical training data

## New: Meal Logging Screen (`/log`)

A dedicated screen for logging any meal, accessible from:
- Home screen "Log a meal" button
- History screen "Add meal" button
- Post-rejection "Cooked something else?" prompt

### Three Input Modes

#### Mode 1: Photo Upload
1. User takes a photo or selects from gallery
2. Photo sent to vision AI (Gemini Vision / Claude Vision)
3. AI returns: dish name(s), confidence, detected ingredients
4. User sees a pre-filled card: "I see: Dal Tadka + Jeera Rice. Correct?"
5. User can edit the dish name(s) or confirm
6. On confirm: meal is logged, photo is deleted (D-047), structured data is stored

#### Mode 2: Free Text
1. User types: "aloo methi, dal fry, and some rice"
2. Text sent to LLM for parsing
3. LLM returns: structured dish names, inferred ingredients, cuisine, category
4. User sees: "Aloo Methi, Dal Fry, Rice — looks right?"
5. User confirms or edits
6. On confirm: if dishes don't exist in DB, custom dish entries are created with AI-inferred metadata

#### Mode 3: Browse Known Dishes
1. User browses/searches the dish database (seed DB + their custom dishes)
2. Multi-select dishes for the meal
3. This is the existing flow, now one of three options instead of the only option

### Post-Logging
- Meal is logged with source: 'manual' or 'photo' or 'text'
- If the dish is new (custom), it's added to the household's dish corpus
- The recommendation engine's candidate pool now includes this dish
- Preference score for logged dishes is initialized at 6.0 (slightly positive)

## Revised Data Model Changes

### Custom Dishes

```
dishes table:
  + is_custom: BOOLEAN DEFAULT false
  + created_by_household_id: UUID (nullable, null for seed dishes)
  + ai_inferred: BOOLEAN DEFAULT false (metadata was AI-generated, not curated)
```

Custom dishes are household-scoped. A custom dish created by household A is not visible to household B (privacy: D-048). The seed DB (is_custom=false) is shared.

### Meal Entry

```
meals table:
  + input_method: 'recommended' | 'manual_browse' | 'manual_text' | 'manual_photo'
  + photo_processed: BOOLEAN (true if photo was used and deleted)
```

### Preferences at Onboarding

```
preferences table:
  + source: 'onboarding' | 'observed' | 'ai_inferred'
```

Onboarding preferences are explicit (user tapped "loves paneer"). Observed preferences come from the learning loop. AI-inferred preferences come from LLM analysis of meal history.

## Revised Recommendation Engine Changes

### Custom Dish Support
- The candidate pool includes both seed dishes AND the household's custom dishes
- Custom dishes with AI-inferred metadata may have less reliable ingredient lists — the engine weights them slightly lower until confirmed by usage (accepted/rejected)

### Seeded History
- If the user logs recent meals during onboarding (Step 4), the engine has immediate context:
  - Repetition check works from Day 1
  - Ingredient usage map is populated
  - First recommendation avoids repeating what was just cooked

### Free Text → Structured Data Pipeline
```
User types: "aloo methi, dal fry, and some rice"
  ↓
LLM parses → [{name: "Aloo Methi", ingredients: [...], cuisine: "North Indian", category: "dry_vegetable"},
              {name: "Dal Fry", ingredients: [...], cuisine: "North Indian", category: "dal"},
              {name: "Steamed Rice", ingredients: ["Rice"], cuisine: "South Indian", category: "rice"}]
  ↓
Check DB: "Aloo Methi" not found → create custom dish
          "Dal Fry" found → use existing
          "Steamed Rice" found → use existing
  ↓
Log meal with dish_ids: [custom-1, dish-26, dish-86]
```

## API Routes

### Existing
- `POST /api/recommend` — recommendation generation (existing, enhanced for custom dishes)

### New
- `POST /api/log-meal` — parse free text or photo, return structured dish data
- `POST /api/recognize-dish` — photo → dish recognition via vision AI
- `POST /api/parse-dishes` — free text → structured dish names + inferred metadata

---

## Implementation Priority

1. **Free text meal logging** — unblocks the learning loop
2. **Expanded onboarding** (steps 2-4) — better Day 1
3. **Photo upload + recognition** — lowest friction logging
4. **Custom dish support in engine** — broader candidate pool
5. **Log a meal screen** — first-class meal logging UI
6. **Post-accept photo step** — bonus learning data
7. **Enhanced history screen** — photos + add past meal
