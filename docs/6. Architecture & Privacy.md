# Architecture & Privacy
## Technical Foundation and Responsible Data Handling

**Version:** 1.0

**Status:** Draft

**Priority:** Critical

**Consolidates:** Chapters 11 & 12

---

# 1. Guiding Constraints

The architecture must satisfy two equally important requirements:

- **Sub-second, offline-capable recommendations** on mid-range smartphones.
- **Privacy by design** — household data is deeply personal and must be treated accordingly.

These constraints are not in tension. On-device computation serves both performance AND privacy.

---

# 2. Privacy Principles

**Minimum Viable Data.** Collect only what directly improves recommendations.

**Observe, Don't Record.** Process and discard raw media. Store extracted structured data, not photographs or recordings.

**Household Ownership.** All data belongs to the household. View, delete, export, full account deletion — all must be supported.

**No Surprise Usage.** If data is used beyond improving that household's recommendations, explicit consent is required.

**No Inference Sharing.** If the system infers sensitive information (health conditions from dietary changes), it must never be shared externally or with other household members without consent.

# 4. On-Device vs. Cloud

| On-Device (Core) | Cloud (Supporting) |
|-------------------|-------------------|
| Recommendation engine | Advanced AI inference (complex NLP, photo recognition) |
| Household Twin (structured layer) | Voice-to-text processing |
| Local inventory management | Population priors & cross-household intelligence |
| Meal history & preferences | Recipe content delivery |
| Basic feedback processing | Weather & external context APIs |
| Offline-capable recommendations | Festival/seasonal calendars |
| | Data backup & multi-device sync |

**Principle:** A household should get excellent recommendations even with no internet connection. Cloud improves quality; it should never be required for basic functionality.

# 5. Recommendation Pipeline

```
1. Context Assembly — Inventory, members present, time, weather, restrictions, events
2. Candidate Generation — Dish database filtered by cuisine profile + inventory
3. Hard Constraint Filtering — Remove restricted, unavailable, allergen-containing
4. Scoring — Preference scores, repetition penalties, effort match, seasonal fit
5. Satisfaction Estimation — Weighted member satisfaction
6. Meal Assembly — Main dish + sides via Regional Cuisine Profile templates
7. Explanation Generation — Human-readable reasoning
8. Confidence Scoring — AI confidence from data quality + model certainty
9. Presentation — Top 1-3 recommendations with explanations
```

**Latency target:** Steps 1-9 in under 500ms on a mid-range smartphone.

# 6. Data Architecture

Local database schema mirrors the Domain Model directly.

**Household Twin Storage:** Structured attributes in local DB. Emergent model as serialized file on device. Model size target: under 5MB per household.

**Per-Household Data Volume:** ~50-100 KB/month. ~1 MB/year.

**Sync:** Bidirectional when online. End-to-end encrypted. Last-write-wins for simple attributes; merge for complex entities.

---

# 7. Data Classification

| Tier | Scope | Examples |
|------|-------|---------|
| Public | Non-sensitive | Cuisine region, general meal categories |
| Household Private | All members, not external | Meal history, inventory, preferences, recommendations |
| Member Private | One member only | Personal health restrictions, individual preference overrides |
| System Internal | AI only, never surfaced | Raw feedback signals, decision context snapshots, training data |

---

# 8. Voice Processing

**On-Device:** Keyword detection, simple command parsing.

**Cloud:** Complex NLU, conversational context, response generation.

**Privacy:** Process locally whenever possible. If cloud required, encrypt in transit, store processed text, delete audio after processing. Users can review and delete voice history. Never use for identity verification without consent.

**Principle:** All voice features must have a touch equivalent. Voice enhances; it never becomes a dependency.

---

# 9. Photo Processing

**On-Device:** Basic image classification (lightweight model). Ingredient detection from fridge photos. Meal recognition from cooked food photos.

**Cloud:** Advanced recognition when on-device confidence is low.

**Data Flow:** Photo → on-device processing → structured data extracted → photo deleted → structured data stored locally.

**Never:** retain raw photographs, use photos for facial recognition, share with third parties.

---

# 10. Consent Model

**At Installation:** Basic data collection, household profile, passive learning. Required for the product to function.

**Progressive Consent:** Camera access, voice processing, shopping integration, cross-household learning — each requires separate consent explaining exactly what data is collected and why.

**Withdrawal:** Users may withdraw any consent at any time. Product degrades gracefully.

---

# 11. Data Retention

| Category | Retention |
|----------|-----------|
| Active data (Twin, history, preferences) | While account is active |
| Temporary data (photos, voice recordings) | Processed and deleted within 24 hours |
| Aggregated anonymized data | Indefinite, with separate consent |

---

# 12. Children's Data

- No individual profiling of children under 13.
- Preferences aggregated at household level.
- No children's data used for cross-household learning.
- Compliance with COPPA, GDPR-K, India's DPDP Act.

---

# 13. Cross-Household Learning

Valuable for cold start but requires strict safeguards:

- No raw household data shared between households.
- Only anonymized, aggregated patterns.
- Minimum 100 household aggregation threshold.
- Explicit opt-in required.

---

# 14. AI Ethics

**Never:** Assume dietary preferences from religion. Assume cooking roles from gender. Assume budget from location. Assume health from age.

**Never:** Create cooking performance anxiety. Shame food choices. Compare households. Use dark patterns.

**Always:** Be honest about limitations. "I don't know enough about your household to recommend confidently" beats a confidently wrong recommendation.

---

# 15. Notification Architecture

Driven by Notification Rules (see *Domain Model* §37). Local scheduler evaluates rules against current state. Max notifications/day: configurable, default 3. Frequency should **decrease** as the Household Twin improves.

---

# 16. Regulatory Compliance

Designed for: India's DPDP Act (2023), GDPR (diaspora users), CCPA/CPRA (US users), COPPA (households with children).

Compliance built into architecture, not added retroactively.

---
